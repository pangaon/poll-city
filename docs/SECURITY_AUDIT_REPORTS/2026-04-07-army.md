# Security Audit Report - 2026-04-07 (ARMY)

**Auditor**: SENTINEL  
**Scope**: IDOR, cross-campaign leakage, middleware bypass, client-side secrets, rate limiting  
**Routes scanned**: 326 API routes  
**Status**: READ-ONLY audit, no code modified

---

## 1. IDOR Vulnerabilities

### CRITICAL: Voice Broadcasts - No Campaign Membership Check

**File**: `src/app/api/voice/broadcasts/[id]/route.ts`

- `GET`, `PATCH`, and `DELETE` handlers authenticate the user and check permission (`sms:read` / `sms:write`) but **never verify the user is a member of the campaign that owns the broadcast**.
- Any authenticated user with `sms:read` permission can read any broadcast by ID.
- Any authenticated user with `sms:write` permission can modify or delete any broadcast by ID.
- **Impact**: Full cross-campaign read/write/delete on voice broadcasts. An attacker can enumerate broadcast IDs and exfiltrate campaign contact lists, modify broadcast scripts, or cancel active broadcasts.

### CRITICAL: Campaigns Customization - No Campaign Membership Check

**File**: `src/app/api/campaigns/[id]/customization/route.ts`

- `GET` checks `settings:read` permission but does **not** verify the requesting user belongs to the campaign identified by `params.id`.
- `POST` checks `settings:write` but similarly lacks membership verification.
- Any authenticated user can read customization data (colors, logo, page views) for any campaign.
- Any authenticated user can increment page views on any campaign (minor but still IDOR).

### MEDIUM: Phone Banking Session PATCH - Missing Ownership Check

**File**: `src/app/api/voice/phone-banking/sessions/[id]/route.ts`

- `GET` and `POST` correctly verify `pbSession.volunteerId !== session.user.id`.
- `PATCH` (end session) does **not** verify ownership - any authenticated user with `canvassing:write` can end any phone banking session by ID.

### LOW: Results Entry - No Campaign Scoping

**File**: `src/app/api/results/entry/route.ts`

- `POST` requires authentication but has no campaign membership check.
- Any authenticated user can submit election result entries for any province/municipality.
- The double-entry verification could be gamed by a single attacker using two accounts.

## 2. Cross-Campaign Data Leakage

### CRITICAL: Voice Broadcasts (same as IDOR #1 above)

Queries in `GET /api/voice/broadcasts/[id]` fetch by primary key with no `campaignId` filter. A user in Campaign A can read broadcast data belonging to Campaign B.

### CRITICAL: Campaigns Customization (same as IDOR #2 above)

`GET /api/campaigns/[id]/customization` returns campaign customization without verifying membership.

### INFO: Well-Protected Routes

The following routes were verified to properly check campaign membership after entity lookup:
- `budget/[id]` - uses `getAuthorisedItem()` with membership check
- `tasks/[id]` - checks membership against `task.campaignId`
- `users/[id]` - enforces `session.user.id === params.id` (self-only)
- `team/[id]` - uses `requireAdmin()` with campaign membership
- `call-list/[id]` - checks membership against entity's `campaignId`
- `volunteers/expenses/[id]` - checks membership + manager role
- `turf/[id]` - checks membership against `turf.campaignId`
- `turf/[id]/stops/[stopId]` - checks membership via turf
- `print/jobs/[id]` - checks membership against `job.campaignId`
- `budget/rules/[id]` - checks membership against `rule.campaignId`
- `volunteers/groups/[id]/members` - checks membership + admin role
- `volunteers/shifts/[id]/checkin` - checks membership against `shift.campaignId`
- `social/consent/[id]` - checks `consent.userId === sessionUserId`

## 3. Middleware Bypass Analysis

**File**: `src/middleware.ts`

### HIGH: Overly Broad Public Path Prefixes

The following paths are public via `startsWith` matching, which grants access to entire subtrees:

| Path Prefix | Concern |
|---|---|
| `/api/v1` | Grants unauthenticated access to ALL `/api/v1/*` routes. Currently includes `/api/v1/approval/[slug]` which is read-only public data, but any new route added under `/api/v1/` will be automatically public. |
| `/api/civic` | All civic endpoints are public including `/api/civic/passport`, `/api/civic/petitions/[id]/sign`, `/api/civic/profile`. The petition sign route requires `apiAuth()` internally, but other civic routes may not. |
| `/api/results` | Makes `/api/results/entry` publicly reachable. The route itself calls `apiAuth()`, but middleware skips the redirect-to-login flow. |
| `/api/webhooks` | `/api/webhooks/vercel` has **no authentication at all** - any caller can POST fake deployment events and spam `operatorNotification` table. |
| `/api/ticker` | Ticker routes use API key auth internally, but middleware bypass means no session check. Acceptable if API key auth is the intended model. |
| `/api/tv` | TV overlay routes are public by design, but confirm all sub-routes should be unauthenticated. |
| `/api/ops/demos` | Listed as public in middleware but the route handler requires `ADMIN`/`SUPER_ADMIN`. Redundant but safe. |

### HIGH: Vercel Webhook Has No Authentication

**File**: `src/app/api/webhooks/vercel/route.ts`

- No signature verification, no shared secret, no IP allowlist.
- Public via middleware (`/api/webhooks` prefix).
- Any attacker can POST fake deployment events to pollute the `operatorNotification` table.
- Could be used for social engineering (fake "deploy failed" notifications).

### INFO: Admin Route Protected

`/api/admin/fix-slugs` is NOT in the public paths list, so middleware requires authentication. The handler additionally checks `SUPER_ADMIN` role. Adequate.

All `NEXT_PUBLIC_` variables are appropriate for client-side exposure:

| Variable | Purpose | Risk |
|---|---|---|
| `NEXT_PUBLIC_ROOT_DOMAIN` | Domain routing | None - public info |
| `NEXT_PUBLIC_APP_URL` | Base URL for links | None - public info |
| `NEXT_PUBLIC_APP_NAME` | Display name | None |
| `NEXT_PUBLIC_VAPID_PUBLIC_KEY` | Web push public key | None - designed to be public |
| `NEXT_PUBLIC_TURNSTILE_SITE_KEY` | Cloudflare Turnstile widget | None - designed to be public |
| `NEXT_PUBLIC_AI_ENABLED` | Feature flag | None |

### MEDIUM: Health Route Leaks Infrastructure Config

**File**: `src/app/api/health/route.ts`

- Public route (no auth) returns boolean flags for which services are configured: `anthropicKey`, `resendKey`, `stripeKey`, `redisUrl`, `turnstileSecret`, etc.
- While it does not leak actual secret values, it reveals the infrastructure stack to attackers (which services are active, which are missing).
- Attackers can use this to focus attack vectors on known integrations.

| 6 | MEDIUM | Health endpoint leaks infrastructure configuration | `api/health/route.ts` |
| 7 | LOW | Results entry: no campaign scoping | `api/results/entry/route.ts` |

### Recommended Fixes (Priority Order)

1. **Voice broadcasts**: Add membership check after `findUnique` in all three handlers (GET/PATCH/DELETE).
2. **Campaign customization**: Add membership check after `findUnique` in GET and POST.
3. **Vercel webhook**: Add signature verification using `x-vercel-signature` header, or move to a secret-bearing path.
4. **Middleware**: Replace `/api/v1` with specific paths like `/api/v1/approval`.
5. **Phone banking PATCH**: Add `pbSession.volunteerId !== session.user.id` check (matching GET/POST).
6. **Health endpoint**: Gate behind authentication, or remove the `envVars` section from the response for production.
7. **Results entry**: Consider adding campaign scoping or at minimum rate limiting.
